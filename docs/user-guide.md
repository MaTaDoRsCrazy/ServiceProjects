# Руководство пользователя

## 1. Назначение системы

Corporate ServiceDesk - это информационная система для учета заявок пользователей и
оборудования организации.

Система решает следующие задачи:

- прием и регистрация обращений сотрудников;
- контроль статусов заявок на обслуживание;
- хранение комментариев и истории обработки;
- ведение реестра оборудования;
- просмотр сводной отчетности по обращениям и технике;
- работа через web-интерфейс, desktop-клиент и мобильное приложение.

## 2. Состав системы

В проект входят следующие компоненты:

- backend API на Java Spring Boot;
- база данных PostgreSQL;
- web-админка для администратора и техника;
- desktop-приложение администратора на Kotlin;
- Android-приложение техника на Kotlin;
- Docker Compose для запуска всех основных сервисов.

## 3. Роли пользователей

### 3.1 Администратор

Администратор может:

- входить в систему;
- просматривать все заявки;
- создавать новые заявки;
- изменять статус заявок;
- просматривать реестр оборудования;
- просматривать отчетную панель;
- использовать web-админку и desktop-клиент.

### 3.2 Техник

Техник может:

- входить в систему;
- просматривать список заявок;
- менять статус заявки;
- добавлять комментарии через API;
- работать через web-интерфейс, desktop-клиент и Android-приложение.

### 3.3 Сотрудник

Сотрудник может:

- входить в систему;
- создавать заявки;
- просматривать только собственные заявки;
- добавлять комментарии к своим заявкам через API.

## 5. Требования для работы

Для полноценного запуска проекта необходимы:

- Docker Desktop;
- доступ к портам `3000`, `5050`, `5432`, `8080`;
- для desktop-клиента - Java 17+ и Gradle или IntelliJ IDEA;
- для Android-клиента - Android Studio и Android Emulator.

## 6. Запуск системы

### 6.1 Запуск через Docker

1. Открыть терминал в корневой папке проекта.
2. Выполнить команду:

```powershell
docker compose up --build
```

3. Дождаться завершения сборки и запуска контейнеров.

Для запуска в фоне:

```powershell
docker compose up --build -d
```

### 6.2 Остановка системы

Для остановки контейнеров выполнить:

```powershell
docker compose down
```

Для остановки с удалением тома базы данных:

```powershell
docker compose down -v
```

## 8. Общий порядок работы

Типовой сценарий использования системы:

1. Пользователь входит в систему.
2. Сотрудник создает заявку.
3. Администратор или техник просматривает поступившую заявку.
4. Ответственный сотрудник переводит заявку в работу.
5. По мере выполнения добавляются комментарии и меняется статус.
6. После устранения проблемы заявка переводится в статус `RESOLVED` или `CLOSED`.

## 9. Работа в web-админке

### 9.1 Вход в систему

1. Открыть страницу `http://localhost:3000`.
2. Ввести логин и пароль.
3. Нажать кнопку `Войти`.

После успешного входа пользователь попадает в административную панель.

### 9.2 Главное меню

В web-админке доступны разделы:

- `Сводка`;
- `Заявки`;
- `Оборудование`;
- `Выход`.

### 9.3 Раздел "Сводка"

В разделе отображаются:

- общее количество заявок;
- количество открытых заявок;
- количество решенных заявок;
- общее количество единиц оборудования;
- диаграмма по статусам заявок.

Этот раздел предназначен для быстрого контроля текущей ситуации.

### 9.4 Раздел "Заявки"

Раздел состоит из двух частей:

- форма создания новой заявки;
- таблица со списком существующих заявок.

#### Создание новой заявки

Для создания заявки необходимо:

1. Ввести тему заявки.
2. Выбрать приоритет:
   `LOW`, `MEDIUM`, `HIGH`, `CRITICAL`.
3. Ввести описание проблемы.
4. Нажать кнопку `Создать`.

После этого новая заявка появится в общем списке.

#### Просмотр списка заявок

В таблице заявок отображаются:

- идентификатор заявки;
- тема;
- приоритет;
- статус;
- автор;
- элемент управления для смены статуса.

#### Изменение статуса заявки

Для изменения статуса нужно:

1. Найти нужную строку в таблице.
2. Выбрать новый статус в выпадающем списке.

Поддерживаются следующие статусы:

- `NEW` - новая заявка;
- `IN_PROGRESS` - в работе;
- `WAITING_PARTS` - ожидание запчастей или внешнего действия;
- `RESOLVED` - проблема решена;
- `CLOSED` - заявка закрыта.

После смены статуса система автоматически отправляет запрос в backend.

### 9.5 Раздел "Оборудование"

В разделе отображаются карточки оборудования.

Для каждой карточки выводятся:

- статус;
- наименование оборудования;
- инвентарный номер;
- тип;
- местоположение;
- ответственный сотрудник.

Кнопка `Обновить` позволяет повторно загрузить список оборудования.

### 9.6 Выход из системы

Для завершения работы нужно нажать кнопку `Выйти`.

После выхода JWT-токен удаляется из локального хранилища браузера.

## 10. Работа в desktop-приложении

### 10.1 Назначение desktop-клиента

Desktop-клиент предназначен для администратора или техника, которым удобнее
работать с заявками вне браузера.

### 10.2 Запуск desktop-клиента

1. Убедиться, что backend запущен.
2. Открыть папку `desktop-admin`.
3. Выполнить:

```powershell
gradle run
```

По умолчанию приложение обращается к API:

```text
http://localhost:8080/api
```

### 10.3 Вход в приложение

На экране входа необходимо:

1. Ввести логин.
2. Ввести пароль.
3. Нажать кнопку `Войти`.

После авторизации открывается рабочее окно.

### 10.4 Основной экран

На основном экране доступны:

- таблица заявок;
- отображение текущего пользователя;
- список выбора нового статуса;
- кнопка `Изменить статус`;
- кнопка `Обновить`.

### 10.5 Изменение статуса заявки

Чтобы изменить статус заявки:

1. Выбрать нужную строку в таблице.
2. Выбрать новый статус из списка.
3. Нажать кнопку `Изменить статус`.

Если строка не выбрана, приложение покажет сообщение об ошибке.

### 10.6 Обновление списка

Кнопка `Обновить` загружает актуальный список заявок из backend.

## 11. Работа в Android-приложении

### 11.1 Назначение мобильного клиента

Android-приложение предназначено для техника и позволяет быстро просматривать
заявки и менять их статус с мобильного устройства.

### 11.2 Подготовка к запуску

1. Запустить backend.
2. Открыть проект `android-app` в Android Studio.
3. Запустить Android Emulator.

Для эмулятора используется адрес backend:

```text
http://10.0.2.2:8080/api
```

### 11.3 Вход в приложение

На стартовом экране необходимо:

1. Ввести логин.
2. Ввести пароль.
3. Нажать кнопку `Войти`.

Рекомендуемый тестовый пользователь:

```text
tech / tech123
```

### 11.4 Экран списка заявок

После входа приложение показывает:

- заголовок раздела;
- кнопку `Обновить`;
- список карточек заявок.

В карточке заявки отображаются:

- номер заявки;
- тема;
- приоритет;
- статус;
- автор заявки.

### 11.5 Быстрые действия

Для каждой заявки доступны кнопки:

- `В работу` - переводит заявку в `IN_PROGRESS`;
- `Решено` - переводит заявку в `RESOLVED`.

После нажатия приложение обновляет список и показывает уведомление.

## 12. Работа через Swagger API

Swagger можно использовать для тестирования функций системы и проверки backend.

Адрес:

```text
http://localhost:8080/api/swagger-ui.html
```

### 12.1 Вход через API

Использовать endpoint:

```text
POST /auth/login
```

Пример тела запроса:

```json
{
  "username": "admin",
  "password": "admin123"
}
```

### 12.2 Основные API-методы

Доступные группы методов:

- `/auth` - вход и получение текущего пользователя;
- `/tickets` - работа с заявками;
- `/equipment` - работа с оборудованием;
- `/reports` - отчетность.

### 12.3 Авторизация в Swagger

После получения JWT-токена его нужно передавать в заголовке:

```text
Authorization: Bearer <token>
```

## 14. Типовые сценарии работы

### 14.1 Создание новой заявки сотрудником

1. Сотрудник входит в систему.
2. Открывает форму создания заявки.
3. Заполняет тему, описание и приоритет.
4. Отправляет заявку.
5. Заявка получает статус `NEW`.

### 14.2 Обработка заявки техником

1. Техник входит в систему.
2. Открывает список заявок.
3. Находит нужную заявку.
4. Переводит ее в статус `IN_PROGRESS`.
5. После выполнения работ переводит заявку в `RESOLVED`.

### 14.3 Контроль руководителем

1. Администратор открывает раздел `Сводка`.
2. Проверяет число открытых заявок.
3. Просматривает распределение по статусам.
4. Контролирует оборудование, находящееся в ремонте.

## 15. Типовые ошибки и способы решения

### 15.1 Не открывается web-админка

Возможные причины:

- контейнер frontend не запущен;
- занят порт `3000`;
- backend не отвечает.

Проверка:

```powershell
docker compose ps
```

### 15.2 Не работает вход

Проверить:

- правильность логина и пароля;
- доступность backend на `http://localhost:8080/api`;
- наличие записи пользователя в базе данных.

### 15.3 Не загружаются заявки

Возможные причины:

- backend не запущен;
- истек JWT-токен;
- база данных недоступна.

Решение:

- выполнить повторный вход;
- проверить состояние контейнеров;
- проверить логи backend.

### 15.4 Android-приложение не подключается к API

Проверить:

- что backend запущен локально;
- что используется адрес `10.0.2.2`, а не `localhost`;
- что эмулятор имеет сетевой доступ.

### 15.5 Desktop-клиент не входит в систему

Проверить:

- доступен ли backend;
- верный ли URL API;
- корректны ли учетные данные пользователя.

## 17. Рекомендации по работе

- использовать разные роли в соответствии с назначением;
- не удалять данные из базы без необходимости;
- регулярно обновлять список заявок при демонстрации;
- при тестировании API сохранять JWT-токен до завершения сеанса;
- для Android-демонстрации заранее проверить запуск эмулятора.

## 18. Заключение

Система Corporate ServiceDesk позволяет продемонстрировать полный цикл работы
информационной системы технической поддержки: прием заявок, их обработку,
ведение оборудования, контроль статусов и работу нескольких клиентских
интерфейсов на едином backend API.
